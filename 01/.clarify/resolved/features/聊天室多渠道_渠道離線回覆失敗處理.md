# 釐清問題

當客服選擇的回覆渠道不可用（如 LINE Reply Token 過期、FB 24hr 窗口關閉、Webchat 用戶離線）時，系統應如何處理？

# 定位

- Feature：聊天室多渠道 - Rule: 開啟聊天室時預設回覆渠道
- Example：「若最近互動渠道離線或無法發送時，前端提示需手動切換其他渠道」

# 多選題

| 選項 | 描述 |
|--------|-------------|
| A | 前端即時提示「此渠道目前無法發送」，自動切換至下一可用渠道 |
| B | 前端提示錯誤，要求客服手動選擇其他渠道（不自動切換）|
| C | 訊息排入佇列，等待渠道恢復後自動發送（最多等 X 小時）|
| D | 發送失敗後記錄日誌，前端顯示「發送失敗」並提供重試按鈕 |
| Short | 提供其他簡短答案（<=5 字）|

# 影響範圍

- 客服聊天室 UI（錯誤提示、渠道切換邏輯）
- 訊息發送 API（錯誤回傳格式）
- 訊息佇列機制（如選擇排入佇列）
- LINE/FB/Webchat Service 層錯誤處理

# 優先級

High - 直接影響客服操作流程和訊息送達率
