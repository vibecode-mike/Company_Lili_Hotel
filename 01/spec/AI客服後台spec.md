# AI 客服後台

## 1. 項目概念框架

### 1.1 項目背景
在不同產業的客服場域中，人工客服約有 80% 的時間重複處理相似問題，導致人力成本高、效率低。
為了解決此問題，本項目將開發一套整合 AI 智能聊天機器人 與 FAQ 後台管理系統 的解決方案，實現智能問答、自動匹配與數據統計，協助企業降低客服成本並提升客戶體驗。

### 1.2 核心目標
- 提供能即時回覆客戶問題的 AI Chatbot
- 打造便於維護的 FAQ 後台管理系統（支援當前產業，將非結構化的企業知識轉化為結構化的數據來引用回答、測試、發布）
- 提升整體客服效率，使人工客服專注於高價值、複雜問題
- 未來可擴充實現 問題命中率與分類統計，協助企業量化常見問題

### 1.3 成功指標
- 人工客服工作量降低 50%
- AI 問答命中率 ≥ 80%（應命中要命中的）
- 知識庫管理效率提升 70%（更新效率）
- 未來可擴充新增至少 5 項客戶未發現的問題，被統計出來加入QA，協助決策

## 2. 用戶分析

### 2.1 目標用戶畫像
- **最終用戶（客戶端）**：使用網站/APP 提問的普通使用者，需求為快速獲取解答
- **客服人員（管理端）**：負責維護知識庫、處理 AI 無法回答的問題，需求為高效的 FAQ 管理工具
- **企業管理者**：關心統計數據與客服效率，需求為洞察客戶關注點並優化決策
- **系統商**：負責產業管理、FAQ 模組授權開通、Token 額度設定

### 2.2 用戶場景分析
- **客戶端場景**：使用者在旅宿業的官網客服入口中提問（Web/LINE/FB），AI 即時回覆常見問題，僅在必要時轉接人工
- **客服端場景**：客服人員可在後台快速新增規則來建立知識庫，未來可擴充至透過統計數據了解哪些問題命中率低、需要補充
- **管理端場景**：管理者透過後台統計報表了解熱門問題、問題分布及客服效率，據此優化業務策略

## 3. 功能需求

### 3.0 項目背景
為開發一套 整合官網前台 AI 智能聊天機器人 與 FAQ 後台管理系統 的解決方案，可針對不同產業去設定該產業的 FAQ 知識庫。系統將所有已啟用的規則（結構化知識資料）作為 prompt context 直接提供給 AI（LLM），由 AI 根據上下文生成自然語言回答。初期以旅宿業為核心場景，藉由「大分類」與「規則」的雙層架構，組織 AI 回答所需的知識內容。

### 3.1 核心名詞定義

**3.1.1 大分類**
- 定義：於特定的產業領域中，用於組織知識庫內容的分類方式
- 範例：訂房 / 設施
- 旅宿業預設 2 個大分類：**訂房**、**設施**

**3.1.2 規則**
- 定義：大分類下的具體知識條目，以結構化欄位呈現，是 AI 生成回答時最細顆粒度的事實依據
- 規則不是問答對，而是結構化的知識資料，直接作為 AI prompt context
- 範例：在「訂房」大分類下，新增一條規則為 { 房型名稱: 豪華雙人房, 房型特色: 海景陽台, 房價: 3500, ... }

**3.1.3 AI 回答機制**
- 系統收集所有已啟用的大分類及規則內容 → 組裝為 AI prompt context → LLM 生成自然語言回答
- 不使用 RAG 向量檢索，直接將規則內容作為上下文提供給 AI
- 反問引導透過 prompt 指引實現（如：AI 判斷資訊不足時主動反問），未來擴充外部 API 調用

### 3.2 核心功能模組

#### 模組一：FAQ 後台管理（大分類列表）

**目的**：管理 AI 聊天機器人引用的知識庫分類，支援大分類啟停、Token 用量監控、語氣設定。

**授權與開通：**
- FAQ 模組採獨立授權機制，由系統商在管理後台為客戶帳號開通
- 若未開通授權，FAQ 模組入口不可用，顯示「請聯繫系統商開通」
- 授權開通與 Token 額度為獨立管理（開通 ≠ 有 Token 額度）

**產業管理（系統商後台）：**
- FAQ 知識庫分產業設定，每個產業都有自己的大分類，初期僅旅宿業
- 系統商可在管理後台新增產業，同時定義該產業的預設大分類及各大分類的結構化欄位
- 客戶後台只看到自己被分配的產業及其大分類

**大分類管理：**
- 每個產業有 2 個大分類，為系統預設，無法經使用者新增、編輯或刪除，系統保留擴充大分類的彈性
- 旅宿業預設大分類：**訂房**、**設施**
- 支援大分類個別的啟用與關閉，關閉後啟用，立即同步前台聊天機器人引用大分類內頁規則；啟用後關閉則無法引用該大分類做回答
- 每個大分類底下可支援多個規則，規則至多 20 個，保留可擴充數量

**Token 用量管理：**
- AI Token 額度由系統商在管理後台設定，客戶僅能查看用量
- FAQ 模組顯示當前 AI Token 可用數量／已消耗數量（純 LLM 對話消耗）
- 若 Token 消耗完畢，立即停用 AI 聊天機器人，自動轉由原系統 CRM 模組「自動回應」的關鍵字回覆（即現有 auto_response 系統）
- 介面給予 AI 回覆鎖定的提示：「當前客服已啟用 自動回應 模組，須加值請聯繫系統商」

**語氣設定：**
- 支援系統全域的語氣調整，語氣共 2 種提示詞（如：專業、真人）
- 更新後即同步於前台聊天機器人的對話語氣風格

**列表顯示：**
- 大分類列表顯示該分類當前已發布的規則數量
- 大分類列表頁面顯示全域的最後更新時間，yyyy-mm-dd hh:mm

#### 模組二：FAQ 後台管理（大分類內頁）

**目的**：可由系統人員訂定每個大分類下的規則新增與編輯，透過預設的表格欄位，將非結構化的企業資訊轉化為 AI 可讀的結構化資料。

**規則結構化欄位：**
- 規則以結構化表格呈現，表格欄位對應當前所選的大分類
- 表格欄位皆為系統預設（由系統商定義），系統使用者僅能依據結構化欄位來新增、編輯、刪除規則內容
- 編輯規則內容時，有字數或欄位為必填的限制，欄位須保留系統的可擴充彈性
- 每個大分類底下可支援多個規則，規則至多 20 個，保留可擴充數量

**旅宿業 - 訂房大分類欄位（共 7 個）：**

| # | 欄位名稱 | 類型 | 必填 | 說明 |
|---|---------|------|------|------|
| 1 | 房型名稱 | text | 是 | 如：豪華雙人房 |
| 2 | 房型特色 | text | 是 | 如：海景、獨立陽台、免治馬桶 |
| 3 | 房價 | text | 是 | 如：3500 |
| 4 | 人數 | text | 是 | 可入住人數 |
| 5 | 間數 | text | 否 | 該房型的房間數量 |
| 6 | url | text | 否 | 外部連結 |
| 7 | 標籤 | tag | 否 | AI 引用此規則回答時，自動貼標至提問者的會員標籤 |

**旅宿業 - 設施大分類欄位（共 7 個）：**

| # | 欄位名稱 | 類型 | 必填 | 說明 |
|---|---------|------|------|------|
| 1 | 設施名稱 | text | 是 | 如：停車場、游泳池 |
| 2 | 位置 | text | 否 | 如：B1、3F |
| 3 | 費用 | text | 否 | 如：免費、200元/次 |
| 4 | 開放時間 | text | 否 | 如：06:00-22:00 |
| 5 | 說明 | text | 否 | 設施詳細描述 |
| 6 | url | text | 否 | 外部連結 |
| 7 | 標籤 | tag | 否 | AI 引用此規則回答時，自動貼標至提問者的會員標籤 |

**規則狀態管理：**
- 每條規則存在 3 種狀態：**測試(未發佈)**、**已啟用**、**已停用**
- 規則經新增或編輯後，為「測試(未發佈)」狀態，僅測試環境能引用此規則，不會立即影響前台聊天機器人的引用
- 規則為「已啟用」狀態，該條規則立即同步至前台聊天機器人，供前台聊天機器人回答時引用
- 支援規則個別的啟用與關閉

**標籤自動貼標：**
- 新增規則時，每一個規則可獨立設定「標籤」欄位
- 當民眾詢問 AI，且 AI 引用此規則回答時，自動將該規則的標籤貼至對應的民眾作為會員標籤
- 等同於原系統手動標記的會員標籤（MemberTag）

**測試聊天視窗：**
- 入口：大分類列表頁與大分類內頁底部嵌入懸浮按鈕
- 點擊後進入專屬的測試聊天頁面
- 讀取所有大分類及規則狀態為「測試(未發佈)」+「已啟用」的兩種狀態
- 支援兩種版本的切換來做對話效果比對：
  - **版本一（測試+正式）**：含「測試(未發佈)」＋「已啟用」的兩種狀態規則
  - **版本二（僅正式）**：僅「已啟用」且為當前正式版本的規則
- 前台聊天機器人僅引用狀態為「已啟用」的大分類及規則
- 當任何一筆規則儲存後，即時更新測試聊天視窗且系統給予提示「當前有新的測試規則」

**發佈機制：**
- 「測試(未發佈)」轉換為「已啟用」狀態，須透過測試聊天頁面點擊「發佈」按鈕
- 發佈後將所有測試(未發佈)規則狀態變更為已啟用，立即同步至前台聊天機器人
- 發佈權限透過現有權限系統管理，新增以下權限項目：
  - `faq.view` — 查看 FAQ 大分類與規則
  - `faq.manage` — 新增、編輯、刪除規則 + 使用測試聊天
  - `faq.publish` — 發佈規則至正式環境（僅指派給特定角色/帳號）
- 所有系統使用者可擁有 `faq.view` + `faq.manage` 權限
- 僅被授予 `faq.publish` 權限的帳號可執行發佈操作

**版本控制：**
- 發佈時自動產生版本快照
- 一筆規則最多保留 2 種版本：當前狀態 與 上一個已啟用過的正式發佈版本
- 系統使用者可手動將規則恢復為上一個版本，於該筆規則的編輯彈窗點擊「版本回復」
- 版本回復：該筆規則的編輯欄位從測試(未發佈)版本複寫回上一個已啟用的正式發佈版本

#### 模組三：AI 聊天機器人（串 CRM）

**目的**：除了降載系統客服的常見問題，中長期要能夠提升客戶滿意度，甚至促成成交轉化，且確保問題能夠被解決，引導終端使用者問對問題。

**AI 回答流程：**
```
用戶提問（Web/LINE/FB）
    → 收集所有已啟用的大分類及規則內容
    → 組裝為 AI prompt context（含語氣設定）
    → LLM 生成自然語言回答
    → 能回答 → 回傳答案 + 自動貼標
    → 無法回答 → 轉人工處理
```

**核心功能：**
- 能透過自然語言理解與回答
- 當前 FAQ 後台有設定且狀態為已啟用的規則，直接作為 prompt context 提供給 AI 生成回答
- 由 AI 意圖分析判斷民眾的語意（如：有停車場嗎？）關聯至對應大分類給予回覆
- 反問引導透過 prompt 指引實現：AI 判斷資訊不足時主動反問收集必要資訊（如：「請問入住日期？」），未來擴充可對接外部 API 調用

**自動貼標：**
- AI 引用規則回答的同時，自動將該規則設定的標籤貼至提問者的會員標籤
- 回傳至原系統後台，等同於原系統手動標記的會員標籤（MemberTag）

**轉人工機制：**
- 判斷 AI 無法回答時，將問題自動歸類於人工處理問題
- 於 CRM 打上新訊息標記等人工回覆

**Token 耗盡降級：**
- 當 AI Token 用量耗盡，自動停用 AI 聊天機器人
- 降級至現有 CRM「自動回應」模組的關鍵字回覆（auto_response）
- 前台不再觸發 AI 回覆，改為關鍵字匹配回覆
