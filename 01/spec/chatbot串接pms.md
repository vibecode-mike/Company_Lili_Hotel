## 1. 項目概念框架
### 1.1 項目背景
於飯店網站提供浮動式的 AI 智能聊天機器人對話介面，作為旅客查詢住宿資訊、確認房況並引導完成輸入所有訂房需要的資訊，回傳至訂房系統使民眾能夠直接透過對話進入到最終付款頁面。

### 1.2 核心目標
提供能即時回覆客戶問題的 AI Chatbot，並且可調用 PMS 回傳的即時資料做調用回覆
可以於  FAQ 後台管理系統 啟用便於維護當前最新房況的即時資料（數據可被引用回答、測試、發布）

## 2. 用戶分析
### 2.1 目標用戶畫像
民眾端場景：使用者在旅宿業的官網客服入口中提問，AI 即時回覆，並且完成使用者的資訊搜集（如：幾間幾人房、入住日期、結束日期）以加速訂房流程
系統使用者端場景：
1. 支援系統使用者在後台透過輸入 PMS 串接參數，以啟用該飯店 PMS 的資訊，使 AI 能夠即時引用回覆
2. 系統使用者能夠檢視當前 2 種版本的知識庫內容供 AI 引用：
2-1. 串接 PMS 回傳的即時資料（原大分類表單鎖定，輸入改為來自PMS的動態欄位不可編輯）
2-2.「FAQ 模組後台管理」模組中的大分類表單

## 3. 功能需求


3.2 訂房資訊：問出幾間幾人房、時間區間
AI 須確保收集到以下關鍵參數，判斷當前訂房資訊搜集的滿足條件：
幾間幾人房（e.g. 我要1間雙人房、我要2間四人房）、入住日期、退房日期（入住區間）
> 注意：詢問方式須引導民眾以「我要X間X人房」格式回答，例如「我要1間雙人房」或「我要2間四人房」
若系統搜集訂房資訊的條件不足，AI 在對答過程中持續引導民眾回答 採漸進式提問，e.g. 每輪僅詢問 1-2 個參數，確保體驗順暢，引導民眾以「我要X間X人房」的方式回答，直至幾間幾人房、入住區間等條件皆滿足，系統才進行下一步調用 API 查詢系統是否有符合條件的房型
以該瀏覽器為單位，記憶當前對話的輪數為 5 輪，超過則視為新對話、重新整理也視為新對話，則清除緩存
須另外記憶的內容為：訂房資訊（幾間幾人房、入住區間）、會員資訊（會員名稱、聯絡電話、Email）
判斷該民眾關閉分頁或重新整理，則另外記憶的內容為：訂房資訊（幾間幾人房、入住區間）、會員資訊（會員名稱、聯絡電話、Email）則清除緩存
Session 識別以 browser_key 為單位（瀏覽器指紋），訂房收斂過程維護 booking_context 物件追蹤目前房型選取狀態
Session TTL：20 分鐘（SESSION_TTL_SEC = 1200），逾時由伺服器 GC 自動清除；前端重新整理或 5 輪對話達上限亦視為新 Session
Session 持久化模式：
  - 測試模式（ENABLE_DB=false）：Session 保存於記憶體，20 分鐘 GC 清除，不寫 DB
  - 正式發布模式（ENABLE_DB=true）：訂房完成後對話紀錄與會員資料持久化至 DB，由 booking-save 觸發寫入
多房型支援：民眾可在同一次訂房中選取不同房型組合（e.g. 1間雙人房 + 1間四人房），系統以 rooms[] 陣列格式維護已選房型清單
3.3 調用 API：即時查詢符合條件的空房型

調用「訂房」大分類的 API 時，先判斷當前系統有無串接飯店 PMS，由系統進行房型推薦，推薦的規則定義為：
11-1. 如系統有串接該飯店的 PMS，則以當前有剩餘間數，且房型的可入住人數足以滿足民眾的入住需求作為推薦；
11-2. 如系統無串接 PMS 則以房型分類欄位的可入住人數足以滿足民眾的入住需求作為推薦
每次調用 API 為即時數據檢索，系統將以該飯店的 PMS 作為主數據來源 (如：房型名稱、房價、剩餘間數)，而「FAQ 後台系統管理」作為輔助數據源 (如：根據 PMS 回傳的 房型名稱 匹配對應的房型特色）
如系統讀取該飯店的 PMS 當前無剩餘間數，則由 AI 主動反問是否給予其他房型推薦
其他房型推薦的定義為：
如系統判斷當前所有房型的剩餘間數，不同房型之間的可入住人數可以組合出足夠滿足民眾的入住人數作為推薦
混搭推薦（mixed availability）：當單一房型無法滿足時，系統自動建議多房型組合（e.g. 雙人房x2 可滿足4人入住）
AI 以 rooms[] 陣列傳遞已確認房型，每筆 item 包含 room_type、quantity、adults_per_room
若該飯店的 PMS 無任何房型推薦，則由 AI 簡短回覆並加入引導 e.g.「哎呀，找不到完全符合條件的房型！目前根據您輸入的人數與日期，暫時沒有對應的選項。
建議您可以調整入住日期，或是留下您的資訊，如有空房以利客服專員主動聯繫您！」
如系調用該飯店的 PMS API 異常，系統則自動降級，啟動「FAQ 模組後台管理」作為引用回覆


3.4 房型推薦：前台聊天機器人顯示可選空房型供民眾選取
房型推薦在聊天機器人視窗輸出「房型推薦卡片」，房型推薦的排序依據匹配度由高至低，匹配度的定義為可入住人數 = 民眾的入住人數
當系統無串接該飯店的 PMS 或 PMS 當前為錯誤狀態，前台 AI 所引用的回覆為「FAQ 模組後台管理」的訂房大分類；卡片外觀與 PMS 來源一致，剩餘間數欄位顯示「待確認」（available_count = null），房價不加「參考：」前綴
14-1. 如系統有串接該飯店的 PMS，則顯示房型名稱*、剩餘間數*、即時房價*、房型圖片 url
14-2. 如系統無串接 PMS（FAQ 模組後台管理），則顯示房型名稱*、一般房價*、剩餘間數（顯示「待確認」）*、房型圖片 url
判斷民眾在前台介面的空房型觸發互動，如點擊「空房數量」按鈕 進行空房的選擇，則由系統給予下一步的會員資訊搜集
點選空房數量的值，須依據該飯店的 PMS 或「FAQ 模組後台管理」的系統設定來顯示數量上限（用前端控制數量的上限）
多房型選取：民眾可在房型推薦卡片中分別選取不同房型的間數，系統累計為 rooms[] 陣列後進入下一步
民眾確認房型後，系統呼叫 POST /chatbot/confirm-room（body: rooms[]、checkin、checkout）收斂訂房選取
3.5 會員資訊：引導民眾輸入基本資料
判斷民眾已選取房型數量，系統進行完整的會員資訊搜集並給予文字引導 e.g.「即將完成！接下來您只需要提供以下基本資料，我將為您建立您的預定資訊！[姓名*]、[聯絡電話*]、[Email]」
會員資訊滿足條件的定義：由民眾填寫的 3 個制式欄位皆填寫完畢：姓名、聯絡電話、Email，且欄位有個別基本的格式判斷，若用戶手機格式錯誤，欄位應簡短提示 e.g.「哎呀，電話格式似乎不太對，請確認是 10 位數號碼喔！」則進行下一步訂單確認，跳轉 URL（用前端控制欄位格式 e.g.手機 10 位數字碼）
在本會員資訊欄位下方加上小字提示 e.g.「您的個資僅用於本次訂房聯繫與 CRM 會員服務，請安心填寫。」
將用戶個資與本次對話標籤（如：詢問房型、入住日期）寫入 CRM，並以瀏覽器唯一識別碼 browser_key 作為 Web Key（瀏覽器指紋，由前端產生並隨每次請求帶入）

3.6 訂單確認：民眾點擊 URL 進入結帳頁並完成付款
系統將顯示當前民眾已選擇的房型、房間數量的基本資訊與會員基本資訊做最後確認，並給予文字引導 e.g.「已幫您建立好訂房資訊，可以點此進一步完成付款」，由 AI 提供「立即預訂」按鈕以完成跳轉
正式流程觸發點：民眾點擊「立即預訂」後，系統呼叫 POST /chatbot/booking-save
  - Request body 包含：rooms[]、checkin、checkout、guest_name、phone、email、browser_key
  - booking-save 同時執行：(1) 寫入 CRM 會員資料與對話紀錄、(2) 向 PMS 建立購物車訂單取得 cart URL
  - Response 回傳 cart_url，前端跳轉至該 PMS 購物車頁面
  ※ POST /chatbot/booking-url 為舊版接口（legacy），正式主流程已移除，不納入遷移
在跳轉按鈕下方加上小字提示 e.g.「房價與房況以跳轉後之訂房系統顯示為準」


### 核心功能模組
模組一：官網前台啟用 AI 智能聊天機器人
調用 API 時，若判斷當前系統有串接飯店 PMS，則優先以該飯店的 PMS 作為引用回覆的主要來源，「FAQ 模組後台管理」的規則作為引用回覆的資訊輔助（即時房價、剩餘房間）


調用 API 時，若判斷當前系統無串接飯店 PMS，則自動啟用「FAQ 模組後台管理」的大分類表單作為引用回覆的來源

模組二：串接 PMS 模組
串接目標：支援系統使用者於後台新增 PMS 參數以啟用即時房況、即時房價透過 AI 智能聊天機器人引用回覆。
當前僅開放「FAQ 模組後台管理」中的 大分類「訂房」 具備 PMS 串接的入口
「FAQ 模組後台管理」模組，點擊按鈕「串接 PMS」，能夠由系統使用者透過輸入 PMS 串接參數，以啟用該飯店 PMS 回傳的資訊，使 AI 能夠即時引用回覆
系統人員在點擊「串接 PMS」時，系統必須即時進行一次連線測試；若連線測試不通過，系統禁止儲存該參數，並透過 Toast 顯示具體錯誤原因（如：API Key 無效、IP 無白名單等）
後台管理介面顯示該飯店 PMS 串接的「最後更新時間 yyyy-mm-dd hh:mm」包含 2 種更新機制：
手動保存成功的時間、系統固定每日更新的時間
當 AI 意圖判斷需引用大分類「訂房」做為引用回覆，系統先判斷該飯店有無串接 PMS，當前 PMS 狀態是否為啟用狀態，調用API是否為通，則引用該飯店的 PMS「即時房價、剩餘間數」來做優先回覆
大分類「訂房」的內頁規則，特定欄位，如：即時房價、剩餘間數、可入住人數、跳轉頁面 url，皆由 API 自動帶入，禁止系統使用者手動輸入，且於後台介面顯示該欄位的引用來源為 PMS
系統使用者在「FAQ 模組後台管理」修改大分類內的規則時，如：房型圖片 URL 或房型特色，當系統使用者修改後點擊按鈕保存，系統應檢核對應該飯店的 PMS 房型代碼是否仍有效，若無效應彈出提示並顯示錯誤原因
## 模組二附：FAQ 規則狀態與發佈流程定義
「發佈狀態」Toggle 對應 FAQRule.status 欄位（啟用 / 停用）
規則存在 3 種狀態：測試(未發佈)、已啟用、已停用
停用後重新啟用的行為：
  - 將 Toggle 從停用切回啟用，狀態必須先回到「測試(未發佈)」
  - 不可直接跳回「已啟用」，須重新通過發佈流程（測試環境驗證 → 發佈）才可上線生效
  - 前台聊天機器人在重新發佈完成前，繼續引用上一個已啟用版本

模組三：串接原系統 CRM 會員管理模組
目的：針對高意圖訂房的民眾，進行會員資料搜集與行為標籤化，達成自動化會員開發做資料存儲至原系統 CRM 的會員管理模組
寫入原系統 CRM 的會員管理模組的觸發點為，判斷 3 個必填欄位皆填寫完畢 [姓名]、[聯絡電話]、[Email]，且判斷點擊按鈕「立即訂房」後，系統才執行寫入動作 將資料則存儲至原系統 CRM，並紀錄該筆會員的建立時間 yyyy-mm-dd hh:mm，以及建立來源為 Web 官網
該筆會員資料寫入 CRM 時須同步存儲該次 Session 的完整對話紀錄，以便人工客服接手時掌握上下文
若該會員沒有點擊按鈕「立即訂房」或關閉視窗，則不進行資料存儲
判斷 3 個欄位 [姓名]、[聯絡電話]、[Email] 皆有基本的格式檢查，如：須檢查電話（10 位數）與 Email（含 @），格式錯誤時由 AI 提示修正
由 AI 意圖分析判斷民眾的語意，當前「FAQ 模組後台管理」有設定且狀態為已啟用的規則可被聊天機器人自動引用回覆，如民眾詢問：「有停車場嗎？」，則關聯至大分類「設施」給予回覆（如引用來源為設施：停車資訊規則）
引用「FAQ 模組後台管理」大分類回答的同時，系統自動取得該規則的「標籤」欄位（FAQRule.標籤）並貼至會員（MemberTag），
等同原系統手動標記的會員標籤，屬系統自動觸發，不需人工操作；
  - 「標籤」欄 = FAQRule.標籤 欄位的視覺呈現，由後台維護人員在規則編輯時設定
  - 規則未設定「標籤」時，AI 引用該規則不執行貼標動作
在同一次對話 (Session) 中，若民眾多次詢問同一主題，系統僅回傳一次唯一標籤，避免 CRM 標籤重複堆疊
判斷 AI 無法回答時，則將該問題自動歸類於人工處理問題，並於 CRM 「會員管理」打上新訊息標記等人工回覆
